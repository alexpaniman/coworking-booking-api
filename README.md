# Coworking Booking API

REST-сервис на FastAPI для бронирования рабочих мест и переговорных комнат в коворкинге.

Проект демонстрирует CRUD-операции, JWT-аутентификацию, роли пользователей, PostgreSQL, Alembic-миграции, автоматические тесты и алгоритм подбора слотов с динамической ценой.

## Возможности

- Регистрация и вход пользователей.
- JWT-аутентификация через Bearer token.
- Роли `admin` и `user`.
- Admin CRUD для локаций, комнат, удобств и правил ценообразования.
- Просмотр комнат, локаций и удобств.
- Создание, просмотр, перенос и отмена бронирований.
- Проверка пересечений бронирований.
- Рабочие часы локаций: нельзя забронировать комнату вне `opens_at` / `closes_at`.
- Технический буфер комнаты: `buffer_minutes` блокирует время до и после брони.
- Ограничение длительности брони: от 30 минут до 8 часов.
- Динамическое ценообразование с учетом типа комнаты, правил цены и загруженности.
- `price_breakdown` в ответах: сервис показывает, из чего сложилась цена.
- Рекомендации свободных слотов по времени, вместимости, бюджету, локации и удобствам.
- Smart booking flow: пользователь выбирает период, когда конкретная комната не важна.
- Telegram-уведомления при создании, переносе и отмене брони.
- Docker Compose для запуска приложения и PostgreSQL.
- Alembic-миграции для схемы БД.
- Автоматические тесты через `pytest` и `TestClient`.
- `pylint.txt` с результатом проверки качества кода.

## Технологии

- **FastAPI** — основной web-фреймворк. Он используется для REST endpoint'ов, Swagger-документации, валидации запросов и dependency injection.
- **Pydantic** — схемы входных и выходных данных API. Например, `BookingCreate`, `RecommendationRequest`, `SmartBookingResponse`.
- **SQLAlchemy** — ORM для работы с реляционной БД через Python-модели: `User`, `Location`, `Room`, `Booking` и другие.
- **PostgreSQL** — основная реляционная СУБД. В проекте есть связанные таблицы, foreign keys и many-to-many связь комнат с удобствами.
- **Alembic** — миграции схемы БД. При запуске контейнера выполняется `alembic upgrade head`.
- **Docker Compose** — воспроизводимый запуск на любой системе, включая NixOS: отдельно поднимаются `api` и `db`.
- **JWT / python-jose** — токены доступа для авторизованных endpoint'ов.
- **passlib + bcrypt** — безопасное хеширование паролей пользователей.
- **httpx** — отправка HTTP-запросов в Telegram Bot API.
- **pytest + FastAPI TestClient** — автоматические тесты API без ручной проверки через Swagger.
- **pylint** — статическая проверка качества кода, результат сохраняется в `pylint.txt`.

## JWT-аутентификация

JWT нужен, чтобы пользователь мог один раз войти в систему и дальше обращаться к защищенным endpoint'ам без повторной отправки пароля.

Сценарий работы:

1. Пользователь регистрируется через `POST /auth/register`.
2. При регистрации пароль не сохраняется в открытом виде. Сервис хеширует его через `passlib` и `bcrypt`.
3. Пользователь отправляет email и пароль в `POST /auth/login`.
4. Сервис проверяет пароль и создает JWT access token.
5. Клиент передает токен в заголовке:

```http
Authorization: Bearer <access_token>
```

6. Защищенные endpoint'ы через dependency `get_current_user` декодируют JWT, достают `user_id` и загружают пользователя из БД.
7. Для admin endpoint'ов дополнительно проверяется роль через `require_admin`.

Что защищено:

- `/users/me`
- создание, перенос, отмена и просмотр своих бронирований;
- smart booking;
- рекомендации;
- admin CRUD локаций, комнат, удобств и правил цены.

Роли:

- `admin` управляет справочниками: локациями, комнатами, удобствами, правилами цены.
- `user` создает и управляет своими бронированиями.

## Модель данных

Основные таблицы:

- `users` — пользователи, роли, email, хеш пароля.
- `locations` — филиалы коворкинга, адрес, часовой пояс, рабочие часы.
- `rooms` — рабочие зоны и переговорки, вместимость, базовая цена, буфер.
- `amenities` — удобства: проектор, доска, монитор и т. д.
- `room_amenities` — many-to-many связь комнат и удобств.
- `pricing_rules` — правила динамического ценообразования.
- `bookings` — бронирования, статус, время, цена и `price_breakdown`.

## Основные endpoint'ы

- `POST /auth/register` — регистрация пользователя.
- `POST /auth/login` — вход и получение JWT.
- `GET /users/me` — защищенный endpoint текущего пользователя.
- `/locations` — CRUD локаций.
- `/rooms` — CRUD комнат и рабочих зон.
- `/amenities` — CRUD удобств.
- `/pricing-rules` — CRUD правил динамической цены.
- `POST /bookings` — создание бронирования.
- `GET /bookings` — список бронирований. Admin видит все, user видит только свои.
- `GET /bookings/{booking_id}` — просмотр одной брони с проверкой доступа.
- `PATCH /bookings/{booking_id}/reschedule` — перенос бронирования с повторной проверкой правил.
- `POST /bookings/{booking_id}/cancel` — отмена бронирования.
- `POST /recommendations/booking-options` — подбор свободных вариантов бронирования.
- `POST /smart-booking/options` — умный поиск: группирует доступные периоды и варианты комнат.
- `POST /smart-booking/book` — бронирование выбранного варианта по подписанному `option_token`.

Локации имеют рабочие часы `opens_at` и `closes_at`; сервис не дает забронировать комнату вне этого окна.
Комнаты имеют `buffer_minutes` — технический перерыв между бронированиями.
Минимальная длительность брони — 30 минут, максимальная — 8 часов.

## Алгоритм рекомендаций

Endpoint `/recommendations/booking-options` принимает дату, временное окно, длительность, количество людей, бюджет, нужные удобства и признак необходимости переговорки.

Алгоритм:

1. Отбирает активные комнаты по вместимости, типу, локации и удобствам.
2. Делит временное окно на слоты по 30 минут.
3. Исключает слоты, которые пересекаются с подтвержденными бронированиями.
4. Считает динамическую цену:

```text
price = base_price_per_hour
      * duration_hours
      * room_type_multiplier
      * pricing_rule_multiplier
      * occupancy_multiplier
```

5. Фильтрует варианты дороже `max_price`.
6. Сортирует варианты по score и цене, возвращает до 10 лучших.

В ответах бронирования и рекомендаций есть `price_breakdown`: базовая цена, длительность, множители и итоговая цена.

## Динамическая цена

Цена считается по формуле:

```text
final_price =
  base_price_per_hour
  * duration_hours
  * room_type_multiplier
  * pricing_rule_multiplier
  * occupancy_multiplier
```

Компоненты:

- `base_price_per_hour` — базовая цена комнаты.
- `duration_hours` — длительность брони в часах.
- `room_type_multiplier` — переговорка дороже рабочего места.
- `pricing_rule_multiplier` — правило цены, например пиковые часы.
- `occupancy_multiplier` — надбавка за загруженность комнаты в этот день.

`price_breakdown` помогает объяснить пользователю и проверяющему, почему получилась именно такая цена.

## Умная бронь

Сценарий умной брони двухшаговый:

1. Пользователь вызывает `/smart-booking/options` и получает периоды дня. В каждом периоде есть доступные комнаты, цена, score, breakdown и `option_token`.
2. Пользователь выбирает вариант и отправляет `option_token` в `/smart-booking/book`.
3. Перед созданием брони сервис заново проверяет доступность, буфер, рабочие часы и цену. Если слот уже занят, вернется `409`.

## Telegram

Telegram-интеграция опциональна. Для включения задайте переменные:

```env
TELEGRAM_ENABLED=true
TELEGRAM_BOT_TOKEN=...
TELEGRAM_CHAT_ID=...
```

Сервис отправляет уведомления при создании, переносе и отмене бронирования. Ошибка Telegram не ломает бронирование.

## Сценарий для видео

1. Открыть `http://localhost:8000/docs`.
2. Зарегистрировать admin и user или выполнить `python -m app.seed`.
3. Авторизоваться через `/auth/login`.
4. Создать локацию, удобство, комнату и pricing rule.
5. Создать бронирование.
6. Показать запрет пересекающегося бронирования и работу `buffer_minutes`.
7. Перенести бронь через `/bookings/{booking_id}/reschedule`.
8. Вызвать `/recommendations/booking-options` и показать `price_breakdown`.
9. Вызвать `/smart-booking/options`, выбрать `option_token`, затем создать бронь через `/smart-booking/book`.
10. Показать фрагмент кода `app/services/recommendations.py`, `app/services/pricing.py` или `app/services/smart_booking.py`.
11. Показать `pylint.txt`.
